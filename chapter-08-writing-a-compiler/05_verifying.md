## Verifying the contract

A deployed contract is just bytecode. Anyone can read those bytes off the chain, but bytes are not source: you can't tell at a glance whether the contract does what its author claims. **Verification** closes that gap. It publishes the original Solidity source to a block explorer like Etherscan, which recompiles it and checks that the result matches the bytecode already on-chain. If they match, the explorer shows the readable source next to the contract, and everyone can audit it.

### Why our toy compiler can't be verified this way

Here's an honest wrinkle. Etherscan verifies by recompiling **Solidity source with `solc`** and comparing to the on-chain bytecode. The bytecode our toy compiler emits is correct (it runs, it returns 42), but it is not byte-for-byte identical to what `solc` produces from the same `Greeter.sol`. `solc` adds metadata, a different function dispatcher, optimiser output, and more. So submitting our source against our hand-rolled bytecode would simply fail to match.

That's not a defeat; it's a precise lesson about what verification actually proves. It proves that *a specific compiler, with specific settings, turns this exact source into that exact bytecode.* Our compiler is a different compiler. To get the green "verified" checkmark, we deploy a second copy compiled by `solc` itself.

### Deploying the solc twin

Foundry's `forge` wraps compilation, deployment, and verification into one command. Point it at the same `Greeter.sol`, and it compiles with `solc`, deploys, and submits the source to Etherscan in a single step:

```sh
export ETHERSCAN_API_KEY=<your key from etherscan.io/myapikey>

forge create src/Greeter.sol:Greeter \
  --rpc-url $RPC \
  --private-key 0x$PRIV \
  --broadcast \
  --verify
```

Because `forge` compiled the bytecode itself, the source it submits matches exactly, and verification succeeds. The output gives you a contract address and a link to its verified page on `sepolia.etherscan.io`.

### Verifying something already deployed

If a contract is already on-chain (say you deployed it earlier without `--verify`), you verify after the fact with `forge verify-contract`:

```sh
forge verify-contract <address> src/Greeter.sol:Greeter \
  --chain sepolia \
  --etherscan-api-key $ETHERSCAN_API_KEY \
  --watch
```

`--watch` polls Etherscan until it reports success or failure. The compiler version and optimiser settings must match what was used at deploy time, which is why pinning `solc` and the optimiser in `foundry.toml` matters: change them after deploying and the recompiled bytecode won't match anymore.

### What you end up with

Two contracts on Sepolia tell the whole story:

- **The toy-compiler contract** at `0x7e2b169a181484b5108b7579f3f0c10d40cc7651`. Deployed from bytecode we generated by hand, proven live by calling `greet()` and getting `42` back. This is the showcase: *our* compiler's output, running on real Ethereum.
- **The `solc` twin** at `0xC5e798Bc34e1dd61AA021daeD1Ef4A17a62905A4`. The same source, compiled by the production compiler and verified on Etherscan, so anyone can read the source and confirm the match.

Side by side, they make the point of the whole chapter concrete. A compiler is a function from source to bytecode. We wrote one, ran its output on a public chain, and stood it next to the industrial-strength one to see both the agreement (they behave identically) and the difference (their bytes, and so their verifiability, diverge). That is exactly the boundary this chapter set out to make visible.
